Collections

While developing Switchboard, one of my ultimate goals was to eliminate allocation of new memory for garbage collection. You shouldn't have to worry about something like that just to integrate a plugin. This framework is designed for game development, and should have an absolutely minimal impact on performance. So, it became necessary to develop collections to replace some of the standard .NET collections. When a collection, like a List, grows in size, it must re-allocate an array of internal memory. For many collections the pattern is to allocate a new array, twice the size of the previous array, copy the contents, and discard the old array for garbage collection. Since I would like to eliminate undue garbage collection all together, I created collections that return unused arrays back to a memory pool.

Creating multiple, generic collection classes, I did not want disparate memory pools of arrays of various lengths and types. I realized that I could use object arrays as a sort of universal underlying type to store anything, regardless of collection or type. So, I created the ObjectArrayPool, which can store an infinite amount of object arrays without ever allocating any additional memory, as long as the arrays are all the same length. This class serves as the basis for pooling all of the underlying memory arrays that serve the other collections. ObjectArrayPools are thread safe.

So, after some benchmarking, I chose specific lengths for memory pages that I might use, 32, 64, 128, and 256. The SharedObjectArrayPool is a static class that holds an ObjectArrayPool for each of those lengths. So, when the generic ObjectPool, ThreadSafeObjectPool, or DynamicQueue need to expand their internal memory, they borrow an object array of one of those four lengths from the SharedObjectArrayPool. So, the main place to profile memory pooling is at the SharedObjectArrayPool. You can set a max count on each of the pools kept there. If they fill to that point, any excess arrays will be subject to garbage collection as usual.

One advantage of Switchboard collections over standard collections is that they don't just grow dynamically. They also shrink dynamically. So, as elements are removed from a collection, the memory that was previously necessary to store those items is automatically returned to the shared pool. One disadvantage of using Switchboard ObjectPool or ThreadSafeObjectPool is that they have an inherent maximum capacity of 65536 by design. If you need a collection that stores more elements than that, you will have to find another solution.

DynamicQueue functions as a queue, first-in first-out, that grows and shrinks dynamically to any length by chaining object arrays together like a linked list. Normally I would advise anyone away from ever using a linked list because their real world performance is abysmal. However, this solution is mostly comprised of arrays, with each array linked to the next after every 256 elements. Also, there is no method for iteration over the queue. So, the performance is not really impacted in this case. LogFileWriter uses this collection to queue up log entries before flushing them to disk.

Switchboard has two types of object pool collections. ObjectPool has a ForEach method for iterating over the elements, and can remove specific elements. This was intended for future use on creating a replacement for the standard way of handling events, which will not allocate garbage as events typically do. The ThreadSafeObjectPool is more limited in its function, but it is, as the name implies, thread safe.

Again, none of these collections contribute to garbage collection. They all share their internal memory allocations as object arrays of one of the four lengths kept in the SharedObjectArrayPool.
